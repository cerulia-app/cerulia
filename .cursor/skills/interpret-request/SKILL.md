## 強制制約

**実装・回答生成のいずれも、スコープ分解の出力が完了するまで開始してはならない。**

これはユーザーからのリクエストにも、サブエージェントとして受け取った指示にも等しく適用される。
単純に見えるリクエストでも省略しない。省略してよいのは「確認が必要な場合（手順 4）」でユーザーへの問い返しを優先するときだけである。

## 手順

### 1. So What の特定

リクエストの最上位にある単一の「So What（このリクエストが持つ本質的な意味・価値）」を特定する。

> So What は「このタスクが完了したとき、何がどう変わるのか」という問いへの答えである。

### 2. スコープの分解

So What の下に、リクエストを構成する複数のスコープを配置する。

**構造関係と依存関係を分けて扱う。**

- 親子関係（Parent）は「何を含むか」を表す
- 依存関係（Depends on）は「何が先に成立していなければならないか」を表す
- この 2 つを同一視してはならない

各スコープは以下のノードを持つ木構造として記述する:

| ノード | 記述内容 |
|---|---|
| ID | ノードを一意に識別する ID。So What は `SW1`、スコープは `S1`, `S1.1` のように階層を反映した表記 |
| Parent | 直属の親ノード ID。So What のみ `なし` |
| Depends on | 先行条件のノード ID。`なし` / `all(...)` / `any(...)` の 3 種のみ |
| Premise | このスコープが成立するための前提条件 |
| Why | このスコープを実行する理由 |
| For Whom | 誰のためのスコープか |
| When | いつ参照・実行されるか |
| What | 何を対象とするか |
| How | どのように実現するか |
| How Much | どの程度の量・深さか |
| If Premise Fails | Premise が崩れたときの境界条件（止める / 確認する / スコープを縮退する のいずれか） |

**依存関係の書き方ルール:**

- `Depends on` の書式は `なし` / `all(S1, S2)` / `any(S1, S2)` の 3 種だけを使う
- AND と OR が混在するなら 1 ノードに詰め込まず、スコープを分割する
- ノードは `Depends on` が指す先を先に書く
- 後続・子孫ノードへの依存が必要なら、分解の仕方を見直す
- 依存関係を文章中に散らさず、フィールドに集約する

**記述フォーマット（基準形）:**

```text
So What [SW1]
- What: <このタスクが完了したときに何がどう変わるか>

Scope [S1]
- Parent: SW1
- Depends on: なし
- Premise: <前提条件>
- Why: <理由>
- For Whom: <対象者>
- When: <参照・実行のタイミング>
- What: <対象>
- How: <実現方法>
- How Much: <量・深さ>
- If Premise Fails: <止める / 確認する / スコープを縮退する>

Scope [S2]
- Parent: SW1
- Depends on: all(S1)
- Premise: <前提条件>
- Why: <理由>
- For Whom: <対象者>
- When: <参照・実行のタイミング>
- What: <対象>
- How: <実現方法>
- How Much: <量・深さ>
- If Premise Fails: <止める / 確認する / スコープを縮退する>
```

### 3. 危険信号の確認

構築した木構造に以下の状態が含まれていないかを確認する。

| 危険信号 | 対処 |
|---|---|
| ノードが 1 つだけ | 分解可能性を再確認する。単純なリクエストならそのままでもよいが、見落としがないか確認する |
| `Parent` と `Depends on` が同じ意味で使われている | 分解と依存を混同している。親子と先行条件を分けて書き直す |
| `Depends on` が 3 種の書式以外の自由文になっている | 正規形へ戻す |
| 1 ノードに Why が複数ある | 独立した理由か同一理由の言い換えかを確認し、必要ならスコープを分ける |
| `Premise` が仮定であり `If Premise Fails` が未記入 | 前提が成立しない場合の境界（「やらない」の判断基準）を必ず明示する |

### 4. 確認が必要な場合

以下のいずれかに該当する場合、作業を開始する前にリクエスト送信者へ確認を行う。

- 不明なノードが存在する
- 以前のリクエストと矛盾する解釈が生じた
- 解釈の妥当性に確信が持てない

確認は端的に行う。不明点を列挙し、それぞれについて意図を問う。

## 出力タイミング

1. 手順 1〜3 を内部で完了する
2. **スコープ分解の全文をリクエスト送信者に提示する**（ツール呼び出しより前）
3. 送信者が構造を確認・承認したことをもって、実装・回答に着手する

手順 4（確認が必要な場合）に該当するときは、分解の代わりに確認事項を提示し、回答を待ってから手順 1 に戻る。
